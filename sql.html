SQL

Реляционные базы данных являются одними из наиболее часто используемых баз данных по сей день, и поэтому навыки работы с SQL для большинства должностей являются обязательными.

SQL расшифровывается как Structured Query Language – язык структурированных запросов. Это декларативный язык программирования, специально разработанный для управления и манипулирования реляционными БД. Он предоставляет стандартизированный способ взаимодействия с БД, позволяя выполнять операции, такие как вставка, обновление, выборка и удаление данных. Важным преимуществом SQL является его универсальность: с его помощью можно создавать, изменять и управлять данными в различных реляционных БД, таких как MySQL, PostgreSQL, Microsoft SQL Server и других.

СУБД (Система Управления Базами Данных) – это программное обеспечение, предназначенное для создания, управления и обслуживания баз данных. Оно предоставляет удобный интерфейс для взаимодействия с данными и обеспечивает эффективное их хранение. СУБД позволяют пользователям создавать структурированные базы данных, обеспечивают безопасный доступ к данным, поддерживают транзакции для обеспечения целостности данных и обеспечивают эффективные механизмы поиска и обработки информации.

Существует два типа СУБД:

  • Реляционная система управления базами данных: данные хранятся в отношениях (таблицах).
  • Нереляционная система управления базами данных: не существует понятия отношений, кортежей и атрибутов.

    • Отсебятина:

    В чем разница между SQL и MySQL? SQL — стандартный язык структурированных запросов на основе английского языка, язык реляционной базы данных, который используется для доступа и управления данными. MySQL — система управления базами данных, реляционная СУБД

    Это организованная коллекция информации, структурированная по определенным правилам. В ее основе могут лежать таблицы, взаимосвязанные между собой, что делает хранение, управление и извлечение данных легкими и эффективными процессами. База данных обычно управляется системой управления базами данных. Данные вместе с СУБД, а также приложения, которые с ними связаны, называются системой баз данных, или, для краткости, просто базой данных.

    Реляционная база данных – это разновидность базы данных. В ней используется структура, которая позволяет нам идентифицировать и обращаться к данным в привязке к другим частям данных из БД. Данные в реляционной БД часто организованы в виде таблиц.

    Выделяют 4 языка SQL:

    DDL (data definition language) - язык определения данных. Позволяет создавать БД, описывать их структуру, а также устанавливать правила размещения в них данных. Основные команды:

  • CREATE — создать таблицы.
  • ALTER — добавить, изменить или удалить столбцы.
  • DROP — оператор удаления объектов из базы данных.
  • TRUNCATE — удалить все строки таблицы.

    • DML (data manipulation language) - язык обработки данных. Позволяет манипулировать данными в БД, изменять уже внесенную информацию, сохранять, обновлять и удалять хранимую информацию. Под данную категорию подпадает основная часть всех SQL команд. Основные команды:

  • SELECT — оператор запроса, возвращающий нам необходимые сведения.
  • DELETE — удаляет строки из таблиц.
  • UPDATE — изменяет табличные данные.
  • INSERT — вставляет новые записи.
  • MERGE — смесь операторов UPDATE и INSERT.

    • DCL (data control language) - язык управления данными. Позволяет настроить работу с базой данных путем установки необходимых разрешений и прав СУБД. Основные команды:

  • GRANT — управление доступом к базе данных.
  • REVOKE — отмена выданных привилегий.
  • DENY — отнимает у пользователя возможность выполнять действия.

    • TCL (transaction control language) - язык управления транзациями. Служит для управления изменениями, происходящими в результате DML-запросов, и объединяет эти DML-запросы в наборы транзакций. Основные команды:

  • BEGIN — определяем начало транзакции.
  • COMMIT — фиксируем изменения.
  • ROLLBACK — откатывает все изменения.

    • Отсебятина:

    Транзакции SQL — это единицы работы или последовательности действий, выполненных в логическом порядке.

  • Таблица (сущность) - объект, содержащий набор данных, представляющая собой набор строк и столбцов, где каждая строка представляет отдельную запись, а каждый столбец определяет тип данных для соответствующего поля записи. Таблица отображает сущность из реальной жизни с набором её признаков в виде столбцов и строк, каждая из которых является отдельной записью об этой сущности.
  • Столбец представляет отдельное поле данных или атрибут.
  • Строка представляет отдельную запись или кортеж данных.
  • Ключи используются для уникальной идентификации каждой записи в таблице.
  • Отношения - определяют связи между таблицами и позволяют объединять данные из разных таблиц.
  • Ограничения целостности - устанавливают правила и ограничения для данных, чтобы обеспечить их целостность и согласованность.
  • Индексы - позволяют ускорить доступ к данным, определяя специальные структуры для быстрого поиска информации.

    • Это базовые составляющие РБД

    Запрос - это команда, которая используется для извлечения, обновления, добавления или удаления данных из базы данных.

    Подзапрос - это запрос, который вложен в другой запрос. Подзапрос выполняется сначала, а затем его результат используется в основном запросе. Подзапросы позволяют делать более сложные запросы, комбинируя результаты нескольких запросов.

    Подзапросы бывают некоррелированные и коррелированные.

  • Некоррелированный подзапрос (non-correlated subquery) – это подзапрос, который может быть выполнен независимо от внешнего запроса, и его результат не зависит от строк, возвращаемых внешним запросом. Это означает, что некоррелированный подзапрос выполняется только один раз, и его результат используется в основном запросе для выполнения операции, такой как фильтрация, сравнение или вычисление.
  • Коррелированный подзапрос (correlated subquery) – это подзапрос, который зависит от внешнего запроса и может быть выполнен для каждой строки возвращаемого внешним запросом результата. Такой подзапрос выполняется для каждой строки в основном запросе, используя значения из этой строки в качестве параметров для подзапроса.

    • Отсебятина:

    Коррелированный подзапрос отличается от некоррелированного подзапроса тем, что он выполняется не один раз перед выполнением запроса, в который он вложен, а для каждой строки, которая может быть включена в окончательный результат.

    Ключ - это столбец или набор столбцов в таблице, не имеющие в строках дублирующих значений. Ключи используются для обеспечения уникальности и целостности данных. Виды ключей:

  • Первичный ключ (Primary Key) представляет собой уникальный идентификатор записи в таблице. Этот столбец содержит уникальные значения для каждой строки, и его цель – обеспечить уникальность идентификации записей в таблице. Другими словами, первичный ключ гарантирует, что в таблице не будет дубликатов строк, и каждая строка будет однозначно определена. Первичный ключ не отменяет возможности объявления и других ключей.
  • Уникальный ключ (Unique key) гарантирует уникальность значений в столбце или наборе столбцов. Допустимо множество уникальных ключей в одной таблице. Допустимы NULL значения.
  • Внешний ключ (Foreign key) поддерживает ссылочную целостность, обеспечивая связь между данными в двух таблицах. Внешний ключ в дочерней таблице ссылается на первичный ключ в родительской таблице.
  • Составной ключ (Composite key) - это ключ, состоящий из двух или более столбцов в таблице базы данных.

    • Отсебятина:

    Разница между первичным ключом и уникальным в том, что первичный генерируется самим sql, уникальный может быть создан пользователем для удобства работы с таблицами.

    Ограничения (constraints) используются для указания ограничения на тип данных таблицы. Они могут быть указаны при создании или изменении таблицы. Они предписывают базе данных отклонять введенные данные, если они не соответствуют ограничению. Примеры:

  • NOT NULL - предотвращает наличие NULL значений.
  • NULL - разрешает наличие NULL значений.
  • CHECK - устанавливает условие, которое значения в столбце (или группе столбцов) должны удовлетворять.
  • DEFAULT - задает значение по умолчанию для столбца, которое будет использоваться, если значение не указано явно при вставке новой записи.
  • UNIQUE - гарантирует уникальность значений в указанном столбце (или группе столбцов), но может допускать NULL значения.
  • PRIMARY KEY - гарантирует уникальность значений в указанном столбце (или группе столбцов) и предотвращает наличие NULL значений.
  • FOREIGN KEY - определяет связь между двумя таблицами, обеспечивая целостность ссылочной целевой таблицы.

    • Связи в базах данных — это способ связывать и организовывать информацию в базе данных. Они логически объединяют сущности через внешние ключи и улучшают производительность системы и эффективность хранения данных. Существуют различные типы связей между таблицами:

  • Один к одному (One-to-One) - каждая запись в одной таблице соответствует одной записи в другой таблице. Пример: таблица "Пользователи" и таблица "Профили", где каждый пользователь имеет только один профиль.
  • Один ко многим (One-to-Many) - каждая запись в одной таблице может иметь несколько соответствующих записей в другой таблице. Пример: таблица "Отделы" и таблица "Сотрудники", где один отдел может иметь несколько сотрудников.
  • Многие ко многим (Many-to-Many) - каждая запись в одной таблице может иметь несколько соответствующих записей в другой таблице, и наоборот. Для реализации (решения) такой связи требуется использование дополнительной промежуточной таблицы, которая связывает две основные таблицы. Пример: таблица "Студенты" и таблица "Курсы", где каждый студент может записаться на несколько курсов, и каждый курс может иметь несколько студентов.

    • Соединения (JOIN) - это операции, которые позволяют объединять данные из двух или более таблиц на основе связанных столбцов. Соединения позволяют объединять данные из разных таблиц, чтобы получить нужную информацию для выполнения запроса. Виды соединений:

  • INNER JOIN - возвращает записи, которые имеют соответствующие значения в обеих таблицах. Если нет совпадения, строка не возвращается.
  • LEFT JOIN - возвращает все записи из левой таблицы и соответствующие строки из правой таблицы. Если нет совпадения, для правой таблицы возвращаются NULL значения.
  • RIGHT JOIN - возвращает все записи из правой таблицы и соответствующие строки из левой таблицы. Если нет совпадения, для левой таблицы возвращаются NULL значения.
  • OUTER JOIN - возвращает все записи из обеих таблиц. Если нет совпадения в одной из таблиц, для недостающих значений возвращаются NULL.
  • CROSS JOIN - возвращает декартово произведение записей из обеих таблиц, то есть каждая строка из первой таблицы объединяется со всеми строками из второй таблицы. Для объединения не используются ключи.

    • Подтипы соединений:

  • SELF JOIN представляется в виде объединения таблицы с её же копией.
  • NATURAL JOIN (естественное соединение) основано на всех столбцах, имеющих одинаковое имя и типы данных в обеих таблицах.
  • OUTER EXCLUDING JOIN возвращает все записи из правой и левой таблиц, которые не совпадают.
  • LEFT EXCLUDING JOIN возвращает все записи из левой таблицы без тех, что присутствуют и в правой.
  • RIGHT EXCLUDING JOIN возвращает все записи из правой таблицы без тех, что присутствуют и в левой.

    • Нормализация — процесс организации данных, цель которого избежать дублирования и избыточности. В процессе нормализации данные преобразуют, чтобы они занимали меньше места, уменьшилась вероятность ошибок и аномалий, а поиск по элементам был быстрым и результативным. По правилам нормализации есть семь нормальных форм баз данных. В некоторых случаях попытка нормализовать данные до «идеального» состояния может привести к созданию множества таблиц, ключей и связей. Это усложнит работу с базой и снизит производительность СУБД. Поэтому обычно данные нормализуют до третьей нормальной формы.

  • Первая нормальная форма - не должно быть дубликатов и составных данных, атрибуты простые, содержащие скалярные значения.
  • Вторая нормальная форма - соблюдается 1 НФ, и каждый не ключевой атрибут неприводимо зависит от первичного ключа. Это означает, что признак, который логически не связан хотя бы с одним первичным ключом, должен быть декомпозирован в отдельную таблицу. Другими словами, у каждой записи в базе данных должен быть первичный ключ.
  • Третья нормальная форма - соблюдается 1 и 2 НФ, и каждый не ключевой атрибут нетранзитивно зависит от первичного ключа. Правило требует выносить все не ключевые поля, содержимое которых может относиться к нескольким записям таблицы в отдельные таблицы. То есть, в записи не должно быть столбцов с неключевыми значениями, которые зависят от других неключевых значений.

    • Отсебятина:

  • 1 НФ пример: ФИО человека следует разделить на три отдельных признака. Адрес также разбить на разные поля.
  • 2 НФ пример: в таблице со взятыми напрокат фильмами название фильма функционально не зависит от персональных данных человека. Следовательно, следует отдельно выделить таблицу с фильмами, связанную через внешний ключ с первичным ключом людей.
  • 3 НФ пример: у человека может поменяться семейное положение на один из вариантов (в браке, не в браке и так далее). Таким образом семейное положение нужно вынести в отдельную таблицу со всеми возможными вариантами этого показателя, а одно из полей основной таблицы информации о человеке сделать внешним ключом на таблицу со статусом. Данные не во всех случаях преобразовывают до третьей формы. Иногда для этого придётся создать много небольших таблиц, что усложнит базу и задачу разработчикам.
